## 一、用户需求洞察

### 1.1 目标用户画像

| 维度 | 描述 |
|:---|:---|
| **核心人群** | 创业者（早期 CEO/联合创始人）、管理者（总监/VP）、小团队 Owner |
| **典型场景** | "我需要一个 AI 销售总监"、"我想做一个我自己的分身"、"帮我配一个文案编辑" |
| **技术水平** | 非技术背景为主，不懂 Prompt Engineering，不了解 OpenClaw 配置体系 |
| **核心诉求** | 快速获得一个"好用的、有灵魂的、懂我的业务的"AI 员工 |

### 1.2 现状痛点分析

```
用户当前路径（传统 OpenClaw 配置）：
┌─────────────┐     ┌─────────────┐     ┌─────────────┐     ┌──────────┐
│ 学习配置体系  │ ──→ │ 手写画像文档  │ ──→ │ 逐轮调教打磨  │ ──→ │ 终于可用  │
│  (2-4小时)   │     │  (1-2小时)   │     │  (3-10轮对话) │     │ (总计1天+)│
└─────────────┘     └─────────────┘     └─────────────┘     └──────────┘

理想路径（灵魂赋能产品）：
┌─────────────┐     ┌─────────────┐     ┌──────────┐
│  描述需求     │ ──→ │ 引导式对话    │ ──→ │ 一键交付  │
│  (2分钟)     │     │ (5-15分钟)   │     │ (80-90分)│
└─────────────┘     └─────────────┘     └──────────┘
```

### 1.3 四大核心痛点

| # | 痛点 | 严重程度 | 产品解法 |
|:---|:---|:---|:---|
| **P0** | **配置门槛高**：不懂"五层架构"、七大文档的概念 | 🔴 致命 | 对话引导，用户只需回答问题，系统自动映射到底层架构 |
| **P1** | **质量不稳定**：缺乏方法论，写出的画像平庸、缺乏灵魂 | 🔴 严重 | 内置最佳实践模板 + 方法论注入引擎 |
| **P2** | **耗时太长**：需要与 AI 来回数十轮对话才能打磨到位 | 🟡 显著 | 结构化引导 + "智能追问"缩短到 3-5 轮核心对话 |
| **P3** | **缺乏参考**：不知道"好的 Agent"长什么样 | 🟡 中等 | 内置标杆案例库 + 实时预览 |

## 二、产品定义与核心逻辑

### 2.1 产品 Slogan 候选

- **主推**："给你的 AI 员工，注入灵魂。"
- 备选："5 分钟，从零到一个有态度的 AI 搭档。"
- 备选："你的第一个数字员工，不应该是个复读机。"

### 2.2 两大核心特性（本产品的灵魂壁垒）

> [!IMPORTANT]
> 以下两大核心特性是本产品与所有通用 AI 工具的根本差异。它们共同构成了 **"智能注入"** 能力——让输出的 Agent 画像不是空洞的模板堆砌，而是真正注入了行业级专业方法论的高水平配置。

| # | 核心特性 | 一句话定义 | 详见章节 |
|:---|:---|:---|:---|
| **⭐ 特性一** | **实时方法论搜索引擎** | 识别经典行业岗位 → 联网搜索该领域最佳实践/思考原则/工作方法论 → 自动注入画像 | [第三章](#三核心特性一实时方法论搜索引擎) |
| **⭐ 特性二** | **官方最佳实践库（缓存层）** | 长期运营维护的结构化岗位方法论库，命中即秒级注入，免搜索 | [第四章](#四核心特性二官方最佳实践库) |

### 2.3 核心交互模型：渐进式灵魂锻造

产品的核心是一个 **"渐进式对话 → 智能搜索/缓存命中 → 多版本迭代 → 双模态导出"** 的闭环：

```
                    ┌──────────────────────────────────────────────┐
                    │              🔮 灵魂赋能引擎                   │
                    │                                              │
  ┌──────────┐     │  ┌───────┐   ┌───────────┐   ┌───────────┐  │  ┌──────────────┐
  │  用户输入  │ ──→ │  │ 对话   │──→│ ⭐最佳实践 │──→│ 方法论     │  │  │  双模态输出    │
  │  (自然语言)│     │  │ 引擎   │   │    缓存库  │   │  搜索引擎  │  │→ │              │
  └──────────┘     │  └───────┘   └─────┬─────┘   └─────┬─────┘  │  │ 📄 画像报告    │
                    │       ↑        命中↓缓存    未命中↓联网搜索   │  │ 📁 配置文件包   │
                    │       │      ┌─────▼─────────────▼─────┐   │  └──────────────┘
                    │       │      │   知识融合 → 注入画像      │   │
                    │       └──────┤   (方法论+思考准则+SOP)    ├──→│
                    │              └───────────────────────────┘   │
                    └──────────────────────────────────────────────┘
```

### 2.4 版本迭代逻辑

| 版本 | 触发条件 | 信息密度 | 预期质量 |
|:---|:---|:---|:---|
| **v1.0 骨架版** | 用户首次描述岗位需求（1-2 句话） | 20% | 50 分 — 基础框架 + 占位内容 |
| **v1.5 血肉版** | 系统追问 3-5 个关键维度后 | 60% | 70 分 — 角色性格初现 |
| **v1.8 方法论版** | ⭐ **缓存命中 / 联网搜索完成**，注入行业方法论 | 75% | 78 分 — 专业方法论已就位 |
| **v2.0 灵魂版** | 用户补充了团队背景、业务场景、协作风格 | 85% | 85 分 — 有灵魂、可直接使用 |
| **v3.0 打磨版** | 用户对 v2.0 进行局部微调 | 95% | 90+ 分 — 精品级交付 |

### 2.5 SBC 原理：信息源渠道分类（Source-Based Classification）

> **核心洞察**：一个高质量的 Agent 画像，其信息并非来自单一来源，而是来自 **多渠道的融合注入**。就像招聘一个真人时需要简历、面试、背调、试用期观察，AI 员工的配置信息也来自 5-18 个不同的信息源渠道。
>
> SBC 原理帮助我们：**识别所有可能的信息渠道 → 分类与优先级排序 → 在产品流程中精准调度**。

## 三、核心特性一：实时方法论搜索引擎

> **核心逻辑**：当系统识别出用户描述的是一个"经典行业岗位"（如销售总监、产品经理、HR 主管），自动触发一轮联网搜索，挖掘该岗位在行业中的最佳实践、思考原则、经典方法论和 SOP，然后结构化注入到画像中。

### 3.1 用户意图识别：引导用户描述"理想员工"

在对话 Phase 2 中，系统会专门有一轮"愿景引导"，帮助用户具象化他们心中理想的 Agent：

> **系统话术**：
> "在你心目中，这个岗位上做到**顶级水准**的人是什么样子的？
> 比如：
> - 他应该精通哪些方法论或行业框架？
> - 他做事的风格是什么？（比如：数据驱动？还是直觉型？）
> - 如果你能请到一个行业大牛来做这个岗位，你期望他带来什么？"

**设计意图**：
- 这不只是收集信息，更是在 **激活用户对"理想岗位"的想象力**
- 用户的回答越具体，后续搜索的关键词就越精准
- 即使用户说不出来，光是"我想要一个销售总监"这句话，系统也能识别为经典岗位并启动搜索

### 3.2 岗位识别与分类引擎

```
岗位识别决策树：

用户输入 → LLM 语义分析
    │
    ├─ 识别为"经典行业岗位" (如：销售总监、产品经理、文案编辑、HR主管)
    │   ├─ 检查 → 官方最佳实践库是否已缓存？
    │   │   ├─ ✅ 命中缓存 → 直接注入（秒级，不搜索）→ 进入 v1.8
    │   │   └─ ❌ 未命中 → 触发联网搜索 → 搜索结果结构化 → 注入 v1.8
    │   └─ 标记："经典岗位，方法论注入"
    │
    ├─ 识别为"复合型/创新型岗位" (如：增长黑客、AI运营官)
    │   ├─ 拆解为 2-3 个子领域
    │   └─ 对每个子领域分别搜索 → 融合注入
    │
    └─ 识别为"纯个人分身" (如：CEO分身、我的替身)
        ├─ 跳过方法论搜索
        └─ 侧重性格/决策风格/价值观采集
```

### 3.3 搜索策略设计

当触发联网搜索时，系统执行以下结构化搜索策略：

#### 搜索维度矩阵

| 搜索维度 | 搜索关键词模板 | 期望获取内容 | 注入画像位置 |
|:---|:---|:---|:---|
| **行业最佳实践** | `"{岗位名} best practices" / "{岗位名} 最佳实践"` | 该岗位的黄金标准、行业共识 | Layer 4：方法论工具箱 |
| **经典方法论/框架** | `"{岗位名} framework methodology" / "{岗位名} 工作方法论 模型"` | SPIN 销售法、AARRR 增长模型等 | Layer 4：独门武器 |
| **思考原则/心智模型** | `"{岗位名} thinking principles" / "{岗位名} 思维模型"` | 第一性原理、逆向思维等 | Layer 2：灵魂准则 |

## 四、核心特性二：官方最佳实践库

> **核心逻辑**：长期运营维护一个结构化的"岗位方法论缓存库"。当用户创建的岗位命中已有的缓存条目时，**秒级注入**，不需要等待联网搜索。相当于一个"CDN 加速层"——把高频岗位的方法论预先精炼好，直接套用。

### 4.1 设计理念：缓存+热更新

```
数据流的优先级决策：

用户描述岗位 → 岗位识别
    │
    ├─ 1️⃣ 官方最佳实践库 (Cache Hit) ─── 命中 → 秒级注入 → v1.8

    │
    └─ 3️⃣ 纯对话生成 (Fallback) ─── 纯个性化岗位 → 跳过搜索 → 直接对话采集
        "太个性化了，只能靠对话挖掘"

长期运营闭环：
    联网搜索的优质结果 → 人工审核/精炼 → 沉淀到官方库 → 下次直接命中
```

## 五、对话引擎设计 — "灵魂访谈引擎"

### 5.1 引导式对话流程（6 Phase）

这是产品的核心心脏——一套结构化的引导话术，将用户的模糊想法转化为高质量的配置维度。**与上一版的区别是新增了 Phase 3（方法论搜索/注入轮），这是两大核心特性在对话流程中的落地。**

#### Phase 1：破冰 & 岗位定义（1 分钟）
> **系统**："你好！欢迎来到灵魂赋能。告诉我，你想要一位什么样的 AI 员工？可以是一句话，比如：'我想要一个帮我做调研的助理' 或 '我想做一个我自己的分身'。"

**提取维度**：
- 岗位名称 / 角色定位
- 初步的职责边界

**系统动作**：
1. 根据用户输入，立即生成 v1.0 骨架版预览（实时显示在右侧面板），给用户"即时反馈"的惊艳感。
2. ⭐ **同步启动岗位识别引擎**（详见第三章 3.2），判断是否为经典行业岗位。

---

#### Phase 2：人格锻造 & 愿景引导（2-3 分钟）
> **系统追问清单**（智能选择 3-5 个最关键的）：

| # | 追问维度 | 示例话术 | 映射的底层架构 |
|:---|:---|:---|:---|
| 1 | **性格画像** | "如果 TA 是你的同事，TA 是什么性格？干练直爽？还是温柔耐心？" | → `IDENTITY` 谈吐人设 |
| 2 | **说话风格** | "TA 跟你汇报工作时，是简洁到一句话？还是会耐心解释推理过程？" | → `IDENTITY` Vibe |
| 3 | **价值观底线** | "TA 最不应该做什么？比如：绝不拍马屁？还是绝不自作主张？" | → `SOUL` Boundaries |
| 4 | ⭐ **理想水平** | "在你心目中，这个岗位做到顶级水准是什么样子？精通哪些方法论？" | → 搜索关键词 + 画像 Layer 4 |
| 5 | **标杆对比** | "有没有一个你特别欣赏的人（真实的或虚构的），希望 TA 像 TA？" | → `SOUL` Core Truths |

---

#### Phase 3：⭐ 方法论侦察（后台 5-10 秒 + 用户确认 1 分钟）

> 这是两大核心特性在对话流中的具体落点。在用户回答了 Phase 2 的问题后，系统在后台自动执行。

**系统动作**：
1. 判断岗位类型（经典/复合/个人分身）
2. 查询官方最佳实践库（缓存层）
3. 如未命中缓存 → 触发联网搜索（详见第三章 3.3）
4. 搜索/缓存结果结构化注入 → 升级画像到 v1.8 方法论版

**用户侧体验**（搜索过程中）：
> 系统 UI 显示搜索动效："🔍 正在搜索'销售总监'的行业最佳实践..."

**搜索完成后**：
> **系统**：
> "我帮你找到了这个岗位的行业最佳实践：
> 1. ⚡ **SPIN 销售法** — 顾问式销售经典
> 2. ⚡ **Pipeline 五力模型** — 漏斗管理关键指标
> 3. ⚡ **挑战者销售法** — 教育客户而非迎合客户
> 
> 这些已经注入了画像（右侧预览已更新）。你觉得哪些适合保留？还有想补充的行业方法吗？"

---

#### Phase 4：业务注入（2-3 分钟）
> **系统追问清单**：

| # | 追问维度 | 示例话术 | 映射的底层架构 |
|:---|:---|:---|:---|
| 6 | **公司/团队背景** | "你们公司是做什么的？大概几个人？什么阶段？" | → `USER` 雇主索引 |

> 1. 📌 TA 的 Slogan 是 'xxx'，你觉得够酷吗？

#### Phase 6：导出 & 部署（30 秒）

> "完成！你可以选择导出方式：
> - 📄 **画像报告**：一份完整的 Markdown 画像文档（适合阅读和分享）
> - 📁 **配置文件包**：直接生成 SOUL.md + IDENTITY.md + HEARTBEAT.md + AGENT.md 的完整文件包（适合直接部署到 OpenClaw）"

### 5.2 智能追问策略

不是所有问题都要问。系统需要一个 **"信息密度评估器"**：

```

## 八、技术架构方案

### 8.1 技术栈选型

| 层级 | 技术选型 | 选型理由 |
|:---|:---|:---|

| **LLM 后端** | **Claude 3.7 Sonnet / GPT-4o** | 长文生成质量高、结构化输出稳定 |
| **状态管理** | **Zustand / React Context** | 管理对话状态、版本历史、导出配置 |
| **文件导出** | **JSZip + FileSaver.js** | 前端直接生成 ZIP 包 |
| **Markdown 渲染** | **react-markdown + rehype** | 实时预览画像文档 |
| **部署** | **Vercel** | Next.js 原生部署，全球 CDN |

### 8.3 核心 Prompt 工程设计

系统 Prompt 是产品质量的命门。核心思路是 **"角色 + 知识库 + 结构化输出约束"** 三层叠加：

```
System Prompt 结构：

[角色设定]
你是"灵魂赋能师"，一位顶级的 AI 角色设计专家。你精通数字员工
的五层架构（Profile → Soul → Knowledge → Methodology → Protocol），
能够通过对话引导，帮助用户在 15 分钟内创建一个有灵魂、有态度、
有方法论的 AI 员工配置。

[内置知识库 - 五层架构标准]
{注入 20 号文档的五层架构定义}

[内置知识库 - 标杆案例]
{注入 Cynthia/Black/Summer 的画像精华片段作为风格参考}

[内置知识库 - 灵魂准则模板]
{注入 SOUL.md 的 Core Truths / Boundaries / Vibe 结构}

[输出格式约束]
每次生成画像时，严格遵循以下 JSON Schema：
{
  "version": "1.0",
  "profile": { "name", "gender", "age", "position", "background", "mbti", "slogan" },
  "soul": { "personality_traits": [], "non_robot_declaration", "soul_principles": [] },
  "knowledge": { "professional_base", "employer_understanding" },
  "methodology": { "responsibilities": [], "toolkit": [] },

[对话策略]
1. 第一轮：生成 v1.0 骨架，同时提出 3 个最关键的追问
2. 用户回答后：升级到 v1.5，再追问 2-3 个深层维度
3. 信息充足后：生成 v2.0 完整版，标注可微调的区域

## 十、实施难度评估

### 10.1 总体评估：⭐⭐⭐ 中等（可行性很高）

| 维度 | 难度 | 说明 |
|:---|:---|:---|
| **前端开发** | ⭐⭐ 低 | 标准 Next.js + 对话 UI，技术成熟 |
| **对话引擎** | ⭐⭐⭐ 中 | 核心是 Prompt 设计 + Phase 状态管理 |
| ⭐ **搜索引擎** | ⭐⭐⭐ 中 | 搜索 API 集成 + LLM 结构化提取，技术成熟但需调优提取质量 |
| ⭐ **最佳实践库** | ⭐⭐⭐⭐ 中高 | 技术实现简单（JSON），但**内容生产是持续性工作**，需要运营团队 |
| **LLM 输出质量** | ⭐⭐⭐⭐ 中高 | 需要大量 Prompt 调优确保输出稳定在 80-90 分 |
| **知识库构建** | ⭐⭐⭐ 中 | 需要将现有文档结构化编码为 Prompt 片段 |
| **文件导出** | ⭐⭐ 低 | 前端纯客户端 ZIP 生成，无后端依赖 |
| **部署运维** | ⭐ 极低 | Vercel 一键部署 |

### 10.2 关键风险 & 预案

| 风险 | 概率 | 影响 | 预案 |
|:---|:---|:---|:---|
| LLM 输出格式不稳定 | 中 | 高 | 使用 JSON Schema 约束 + 重试机制 + 后处理格式化 |
| 画像质量低于 80 分 | 中 | 高 | 预置标杆段落自动注入 + 方法论模板库兜底 |
| 搜索结果质量不稳定 | 中 | 中 | 多源交叉 + LLM 质量评分过滤 + 官方库兜底 |
| 官方库初期覆盖不足 | 高 | 中 | 搜索引擎作为备用，搜索结果自动标记“待入库”，快速积累 |
| Token 成本过高 | 低 | 低 | 对话上下文压缩 + 仅在关键节点调用大模型 |

### 10.3 独特竞争壁垒

> [!IMPORTANT]
> 产品的核心壁垒有**三层**：
> 1. **架构壁垒**：数字生命七大文档 + 五层画像架构，这是任何通用 AI 工具无法复制的 Domain Knowledge。
> 2. **⭐ 搜索壁垒**：实时联网搜索 + 结构化提取，让每个画像都带有行业级方法论。
> 3. **⭐ 数据壁垒**：官方最佳实践库随时间累积像滚雪球，缓存层越厚，用户体验越好、速度越快。

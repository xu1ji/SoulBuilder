# 42. Skill：从入门到高阶全量指南

本指南定义了 OpenClaw 数字员工如何通过 **Skills (技能脚本)** 获取“肌肉记忆”与“业务算法”。OpenClaw 的 Skill 系统遵循 **AgentSkills 开放标准**，本质是用自然语言定义的“操作手册”。

如果说 MCP 是外部设备的标准插槽，那么 Skill 就是**AI 学习并固化下来的特定动作流**。它让 AI 能够像专家一样处理复杂任务，同时通过 **热加载 (Hot Reload)** 机制实现功能的即时生效。

---

## 📖 第一部分：Skill vs MCP——深度对比

| 维度 | Skill (技能) | MCP (上下文协议) |
| :--- | :--- | :--- |
| **本质** | **业务逻辑的封装**。通常是 Prompt + 脚本。 | **技术能力的标准化插槽**。JSON-RPC 协议。 |
| **运行位置** | **执行层**（Agent 决策后调用）。 | **基础设施层**（Gateway 层面挂载）。 |
| **安装优先级**| **工作区 > 全局 > 内置** (Workspace > Local > Bundled) | 统一由配置文件驱动挂载。 |
| **擅长领域** | 处理具有特定步骤、需要逻辑核查的业务流。 | 连接标准的外部服务（如数据库、GitHub）。 |
| **形象比喻** | 像是一个**“操作手册”**，教 AI 如何一步步干活。 | 像是一个**“USB 接口”**，给 AI 插上外部硬件。 |

### 2. 技能在架构中的运转模型
1. **中枢 (Gateway)** 收到消息，交给 **大脑 (Agent)**。
2. **大脑** 基于 LLM 推理，决定是否需要使用工具。
3. **技能 (Skills)** 被唤起并执行实际动作（如：读文件、发邮件）。
4. **结果回传** 给大脑，大脑整理后返回给用户。
**核心逻辑：模型负责“想”，Skills 负责“干活”。**

## ⚙️ 第二部分：三阶进化教程——从小白到开发者

### 【入门级】开箱即用：激活“核心工具”与“官方套装”
新版 OpenClaw 区分了**原子级工具 (Core Tools)** 与 **官方捆绑技能 (Official Bundled Skills)**：

#### 1. 核心工具 (必须掌握的“手脚”)
这是系统底层的原子能力，共约 **25 个**，决定了 Agent 能否与物理世界交互：
*   **exec/bash**: 执行 Shell 命令（安装、运维、脚本运行）。
*   **read/write/edit**: 极高精度的文件系统操作。
*   **browser**: 基于 Playwright 的全功能浏览器模拟。
*   **web_search/web_fetch**: 语义级的网页搜索与信息抓取。

#### 2. 官方捆绑技能 (预装的“大脑插件”)
系统自带约 **53 个** 全量内置技能，默认全部启用（受 `skills.allowBundled` 配置控制）：
*   **信息处理**: `web-search` (Brave Search 支持), `weather` (实时天气)。
*   **Apple 生态**: Notes, Reminders, iMessage, Bear Notes。
*   **Google Workspace**: Gmail, Calendar, Drive, Docs, Sheets。
*   **效率重器**: Slack, GitHub CLI, Linear, Cron, **document-tools** (PDF/Excel 高精度转换)。
*   **AI 增强**: Whisper, Python REPL, **coding-agent** (产生 Claude Code 子进程)。

### 【进阶级】社区生态：像刷应用商店一样扩展
通过 **ClawHub (5000+ 社区技能)** 快速武装你的 Agent。
*   **安装命令**：在 OpenClaw 终端执行 `npx clawhub install <skill-name>`。
*   **热加载**：安装完成后，无需重启，AI 会在对应的 Context 下自动感知新技能。
*   **发现渠道**：关注 `smithery.ai` 或 `mcp-get.com`（部分 Skill 以 MCP 形式存在，但逻辑互通）。



### 【高阶级】自主编写：定义你的“企业专有技能”
Skill 通常存放在 `~/.openclaw/skills/` 或 `<project>/skills/` 下。

#### 1. 开发思维：Input -> Action -> Output
*   **输入 (Input)**: 结构化 Schema，告诉 AI 需要哪些参数。
*   **动作 (Action)**: 实现业务逻辑（调 API、写文件、甚至调用另一个 Skill）。
*   **输出 (Output)**: 返回给 Agent 的结果，用于下一步决策。

#### 2. 技术落地：Skill 开发清单
一个标准技能文件夹通常包含：
*   **skill.json**: 定义触发词（intents）、元数据及输入输出约束（Schema）。
*   **SKILL.md**: 核心操作指南。必须包含：
    *   **YAML 元数据**（声明权限与变量）。
    *   **步骤清单**（拆解 atomic actions）。
    *   **验证与故障排除 (Troubleshooting)**：说明如何确认操作成功，以及遇到常见错误（如格式不兼容）时的自动修复逻辑。
*   **Knowledge/** (可选): 存放品牌规范、固定模版等知识资产。
*   **index.ts/js**: 核心执行代码。
*   **skill-tester**: 内置的自动化验证工具。

## 🛠️ 第五部分：防坑与最佳实践

### 1. 避免“技能肥大症” (Token Overload)
*   **原则**：Skill 描述越精炼越好。每增加 100 字，AI 的单次调用成本就会上升。
*   **方法**：将复杂的“指令”部分转移到脚本中执行，`SKILL.md` 只保留顶层流程。

### 2. 沙盒安全原则 (Sandbox First)
*   永远不要给未知的社区 Skill 全局 `sudo` 权限。
*   在 `SKILL.md` 中强制使用相对路径，避免 AI 误操作宿主机根目录。

### 3. 如何解决技能冲突？
如果安装了多个同类技能（如两个 Google Search），AI 可能会“犹豫不决”。
*   **解决方案**：在神经总线的 `skills` 声明中，通过 `priority` 字段明确指定优选级。

## 💎 第六部分：专家级“配置瘦身”与系统优化 (Pro Tips)

### 1. 使用 Meta-Prompt 为配置文件瘦身
*   **痛点**：随着项目复杂度增加，`CLAUDE.md` 或 `OPENCLAW.md` 往往突破千行，导致 Token 浪费。
*   **方案**：使用 **Meta-Prompt** 分析主配置，提取具体的操作流程（如“生产部署”、“数据库迁移”）并封装到独立的 Skill 中。
*   **效果**：主文件行数可减少 **29% 以上**，仅保留对 Skill 的引用。

### 2. Skills vs Slash Commands (斜杠命令)
*   **Slash Command**：用户手动触发，适合快捷操作、单一指令。
*   **Skill**：**AI 自动决定调用**，适合专业任务、多步流程，且具备自我修正能力。

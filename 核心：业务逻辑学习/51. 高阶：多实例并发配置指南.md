# 🦀 51. 高阶：多实例并发配置指南 (Swarm 群落之书)

> [!NOTE]
> 本文档是基于 MacMini (M4 系列) 硬件能力、Docker 虚拟化技术以及 OpenClaw 数字生命架构的深度整合方案。旨在实现从“单兵作战”到“群落协同”的科学跨越。

---

## 🎯 一、 目标与愿景：从“大龙虾”到“大龙虾群落” (Swarm)

你现在的目标不仅是跑通一个 OpenClaw，而是构建一个**科学、高效且安全**的 AI 专家团队（2 到 20 个实例）。

*   **愿景**：构建一个由不同灵魂 (SOUL)、不同人设 (IDENTITY) 和不同技能 (TOOLS) 组成的 AI 专家团队，实现 24/7 的自主协同作业。
*   **科学分工**：给不同项目配独立实例（不同权限/技能）、AB 测试不同模型组合、隔离高风险任务（如爬虫/自动交易）。

---

## 🔬 二、 可行性分析：硬件天花板与现实边界

### 1. 内存：真正的“隐形刺客” (RAM Analysis)
根据 `260216：硬件能力验证分析.md` 与多篇实战报告：
*   **物理内存**：每个 OpenClaw 服务及 Node 进程约占 300-500MB，但 **浏览器 (Playwright/Chromium)** 是最大变量，每个标签页活跃时可达 512MB-1GB。
*   **承载配比 (32G/64G)**：
    *   **32GB 内存**：建议分配 20-22GB 给 Docker。可稳定运行 10-15 个实例，但需严格限制并发（详见后文）。
    *   **64GB 内存**：可支撑 20+ 个实例，允许 2-5 个核心实例同时进行重度浏览器操作。

### 2. 性能瓶颈 (CPU & I/O)
*   **CPU**：M4 Pro 的 12 核架构可灵活调度 10-20 个实例。
*   **存储**：20 个实例的 Workspace 读写频繁，**强烈建议外接 NVMe SSD** 存储模型与工作区数据，防止宿主机磁盘挤兑。

---

## ⚙️ 三、 基本原理：“共享大脑，独立身体”

为了实现资源的最优利用，我们采用 **“1 + N” 科学架构**：

1.  **1 个共享大脑 (Logic Core)**：
    *   在宿主机 (macOS) 直接运行 **Ollama**。
    *   所有 20 个容器通过 `host.docker.internal:11434` 共享本地 14B 模型 (如 Qwen2.5-14B)。这比每个容器内跑大模型节省了 90% 的 VRAM。
2.  **N 个独立身体 (Lobster Bodies)**：
    *   使用 **Docker Compose** 进行容器化编排。
    *   每个身体挂载独立的 `.openclaw` 路径，确保每个人的“记忆、Soul、Identity”互不干扰。

---

## 🏗️ 四、 路线选择：逻辑多开 vs 物理隔离

在决定部署 20 个实例前，请先明确你的需求场景：

*   **路线 A：逻辑多开 (轻量)**：
    *   在一个 OpenClaw 实例内连接多个 Channel（钉钉/飞书）并定义多个 Persona。
    *   *适用*：任务背景重合、仅需身份名称不同的简单助理。
*   **路线 B：物理多开 (推荐)**：
    *   通过 Docker 或 Native 分身，拥有独立端口和文件系统。
    *   *适用*：需要 **强隔离、独立文件权限、多租户** 或高风险业务。

---

## 🚧 五、 核心难点：端口矩阵、API 限流与安全权限

1.  **端口冲突 (Port Map)**：
    *   每个大龙虾需独立 Web Dashboard 端口。建议采用业务分段：
        *   `1880x` 系列：财务/机密组。
        *   `1881x` 系列：研发/内容组。
2.  **API 速率限制 (Rate Limiting)**：
    *   20 个实例共用一 key 易触发 `429` 报错。在 `11. JSON` 中分配多 Provider 负载均衡。
3.  **权限分级 (L1-L3 Security)**：
    *   **L1 (外部)**：仅限 RAG 检索，禁止工具调用。
    *   **L2 (内部)**：允许 Web 搜索/文件总结，**写操作需人工回显确认**。
    *   **L3 (本人)**：Truman 专属，拥有 Shell 权限与密钥访问权。

## 🛠️ 六、 具体配置操作 (Docker 编排版)

### 1. 硬件与系统预备
*   **Docker Desktop**：Resources 设为“CPUs: 80%, Memory: 24GB+”。
*   **Colima (可选)**：针对追求极致性能/稳定的用户，可作为轻量容器运行时。
*   **环境变量管理**：创建根目录 `.env` 集中存放 API Keys，防止泄露。

### 2. 构建“群落”目录结构
```bash
~/AI_Team/
├── .env                  # 密钥库
├── docker-compose.yml    # 编排中心
└── nodes/
    ├── node_01/
    │   ├── config/       # 独立配置 (openclaw.json)
    │   └── workspace/    # 独立记忆与文件
    └── ...
```

### 3. 一键批量生成脚本 (自动化)
```bash
# 执行此循环可快速生成 20 个逻辑隔离的“家”
for i in {01..20}; do
  mkdir -p nodes/node_$i/config nodes/node_$i/workspace
done
```

### 4. `docker-compose.yml` 模板 (科学编排)
```yaml
version: '3.8'
services:
  writer-agent:
    image: ghcr.io/openclaw/openclaw:latest
    ports: ["18801:18789"] # 通过端口偏移区分
    deploy:
      resources:
        limits:
          memory: 4G # 给核心写作员更多内存
    environment:
      - OPENCLAW_AUTH_TOKEN=${TOKEN_01}
      - OPENAI_API_KEY=${OPENAI_API_KEY}
    volumes:
      - ./nodes/node_01/config:/root/.openclaw
      - ./nodes/node_01/workspace:/root/.openclaw/workspace:rw
    restart: always

  assistant-20:
    image: ghcr.io/openclaw/openclaw:latest
    ports: ["18820:18789"]
    deploy:
      resources:
        limits:
          memory: 1G # 边缘助理限制资源
    # ... 其他环境配置
```

## 🛡️ 七、 安全加固与高级维护

1.  **Launchd 守护**：配置 `launchd` 脚本，确保 Mac 重启后 Docker Swarm 自动后台恢复运行。
2.  **Tailscale 组网**：**严禁将 188x 端口映射到公网**。通过 Tailscale 实现远程设备与这 20 个实例的私有连接。
3.  **灾备备份 (Rsync)**：
    *   建议每日凌晨：`rsync -av --delete ~/AI_Team /Volumes/Backup_SSD/`。
4.  **影子虚拟机 (高风险隔离)**：
    *   对于处理“黑灰产”或“实验性高危插件”的实例，不建议放进 Docker，应放入 **UTM/Parallels 独立的 Ubuntu 虚拟机**。

---

## 💡 八、 最后的专家提示 (Pro Tips)

*   **11. JSON 适配**：必改 `maxConcurrent`。在多实例并行下，单个实例设为 **1** 最为稳妥，防止挤兑宿主机内存。
*   **不迷信物理隔离**：实例间的隔离是物理的，但如果你的 `IDENTITY` 配置不当，AI 依然可能通过指令诱导产生误操作。**安全是一个持续更新 `SOUL.md` 的过程。**
*   **监控**：定期在宿主机使用 `htop` 查看 `Chromium` 进程数量，它是系统不稳定的唯一红线。

* Heartbeat：循环（主会话）
* Cron(main)：定时（主会话）
* Cron(isolated)：纯独立（没有上下文）

## 1. 三种机制总览：记忆 & 会话位置 对照表

结合你的补充表格，我们把 Heartbeat / Cron(main) / Cron(isolated) 按三个维度统一整理：

### 1.1 记忆 & 会话位置 & 机制

| 机制类型           | 是否共享历史记忆 | 会话位置               | 会话机制说明                                                                 |
|--------------------|------------------|------------------------|------------------------------------------------------------------------------|
| **Heartbeat**      | ✅ 共享历史       | 永远在 **主会话**      | 定时触发，作为 system event 注入主会话，读取 `HEARTBEAT.md`，有记忆、有上下文 |
| **Cron (main)**    | ✅ 共享历史       | **主会话** 内          | 按 Cron 表达式定时触发，任务消息作为系统事件注入主会话，能看到对话和记忆      |
| **Cron (isolated)**| ❌ 不共享历史     | **独立会话（session 外）** | 每次 Cron 触发新建独立会话，执行完即销毁，不读主会话历史，不写主会话历史     |

> 这里的“共享历史”指：  
> 能否看到主会话的聊天记录、`USER.md`/`MEMORY.md`/近期 `memory/YYYY-MM-DD.md` 等被注入到提示词中。  
> 实际上，文件类记忆机制在主会话启动时是强制读取的（AGENTS.md 规定），独立会话更像是“只注入极少量系统提示 + 专用任务说明”的精简版本。

### 1.2 图解：主会话 & 独立会话

你原来的示意已经很准确，这里用文字再解释一次，方便你以后扩展：

#### 主会话（带长期记忆）

```text
┌─────────────────────────────────────────┐
│               主会话                   │
│                                         │
│  [你的消息]  [Agent 回复]               │
│  [你的消息]  [Agent 回复]               │
│  [Heartbeat 系统事件] → [心跳检查]       │  ← Heartbeat：永远在这里跑
│  [Cron(main) 系统事件] → [处理]         │  ← Cron(main)：也在这里，能看历史
│                                         │
└─────────────────────────────────────────┘
```

特点：

- 会话历史持续存在（聊天 + 工具调用 + 文件更新痕迹）
- 每次 Heartbeat / Cron(main) 执行，都可以利用这些历史 + 记忆文件做“聪明”的判断

#### 独立会话（隔离执行）

```text
┌─────────────────────────────────────────┐
│               独立会话                 │
│                                         │
│  [Cron 任务消息] → [执行] → [发送结果]  │  ← Cron(isolated)
│                                         │
└─────────────────────────────────────────┘
↑
每次 Cron 触发都新建
任务结束就销毁（不留历史）
```

特点：

- 默认不带主会话历史，也不会回写主会话历史
- 更像是“为某个定时任务临时开的单独 agent 实例”

## 2. 三种机制的精确理解 & 适用策略

### 2.1 Heartbeat：带记忆的“后台巡检”

- **永远运行在主会话中**（文档明确说明）
- 每次触发时：
  1. Gateway 发一个 `heartbeat` 事件到主会话
  2. Agent 读取 `HEARTBEAT.md`，执行里面定义的任务
  3. 一切正常 → 输出 `HEARTBEAT_OK` → 网关抑制，不推送给你
  4. 有异常 / 重要变化 → 正常回复消息，你能看到

**适用原则：**

- 需要“持续监控 + 只有异常打扰你”的任务
- 需要读写长期记忆，如：
  - 把异常写进 `memory/2026-02-23.md`
  - 根据 `MEMORY.md` 的长期知识优化策略
- 不适合用来做“每次都给你日报”的任务（会信息泛滥）

---

### 2.2 Cron(main)：带记忆的“定点触发”

- 也是在主会话里执行，只是**触发机制**是 Cron，而不是短周期心跳
- 与 Heartbeat 不同点：
  - **是否静默**：Cron(main) 一般默认会把结果发给你，没有 HEARTBEAT_OK 静默约定
  - **语义角色**：Heartbeat 偏系统健康/后台维护，Cron(main) 偏“你设的闹钟/提醒/定时任务”

**适用原则：**

- 需要：
  - 在具体时间点触发（每天 9 点、每周一 10 点…）
  - 又需要结合主会话历史做判断/对比/个性化汇报
- 典型场景例如：
  - 周报草稿：汇总本周 `memory/*.md`、`learnings/*.md`，生成一份周报草稿
  - 每日“回顾 + 第二天计划”：盘点当天对话和任务完成情况，帮你规划明天

---

### 2.3 Cron(isolated)：无记忆、一次性“干完就走”

- 每次触发：
  - 新建一个独立会话（只有必要的系统提示 + 当前任务指令）
  - 完成任务 → 把结果通过某个通道发给你 → 会话销毁
- 不读取主会话历史，也不向主会话写入任何聊天消息

**适用原则：**

- 任务逻辑固定、对上下文不敏感
- 不希望污染主会话历史（否则主会话会被一堆报表刷屏）
- 对成本敏感，想用短提示词跑轻量模型

## 3. 你的当前 Cron 配置：属于 Cron(isolated)

> 你现在的 Cron 配置：  
> Cron (isolated) → 每 15 分钟  
> ↓  
> 新建独立会话 → 检查项目 → 汇报给你  
> ↓  
> 会话销毁（不留痕迹）

结合上面的机制：

- **是的，这就是 Cron(isolated）典型模式**
- 每 15 分钟都会：
  - 在“会话外”跑一轮检查
  - 不记得上一次检查的结果（除非你让它自己记在外部文件/系统）
  - 主会话里不会自动累积这些执行历史

## 4. 更多实战场景：按三种机制分别给你大量可复用案例

下面所有场景会标明适合用 Heartbeat / Cron(main) / Cron(isolated)，你可以直接挑选组合。

### 4.1 Heartbeat 场景（主会话 + 有记忆 + 静默为主）

#### 场景 A：服务器 / 服务健康巡检

- **机制**：Heartbeat
- **行为**：
  - 每 X 分钟：
    - Ping 指定 API/URL
    - 检查日志错误率、CPU、磁盘空间（通过自定义工具）
  - 正常 → `HEARTBEAT_OK`，不打扰你
  - 异常 → 发简短告警，并写入 `memory/2026-02-23.md`
- **优势**：
  - 长期记住：这台服务最近老出问题？频率变高了吗？  
  - 可以在周末再根据这些 daily memory 做一个故障周报（用 Cron(main)）

#### 场景 B：记忆系统自我维护

- **机制**：Heartbeat
- **行为**：
  - 定期检查：
    - `memory/` 目录是否过大
    - `MEMORY.md` 是否需要压缩 / 去重
  - 自动进行：
    - 老日志归档
    - 长期记忆简化（提炼成要点、合并重复）
- **优势**：
  - 控制 Token 成本（记忆文件过大时自动压缩）
  - 长期主会话不被“垃圾记忆”拖慢

#### 场景 C：长期任务的“状态守护者”

比如多天跨度的大项目、学习计划：

- **机制**：Heartbeat
- **行为**：
  - 每天/每隔几小时：
    - 检查 `PLAN.md`、项目任务列表
    - 跟踪哪些子任务长期未更新
  - 正常 → `HEARTBEAT_OK`
  - 如果发现计划长期未推进 → 给你发一条“温柔催促”
- **优势**：
  - 能利用 `MEMORY.md` / `memory/*.md` 来理解：你最近是不是被别的任务占满了？
  - 建议调整计划，而不是机械提醒

---

### 4.2 Cron(isolated) 场景（无记忆、标准化、报表类）

下面这些都非常适合你当前的“Cron 每 15 分钟 / 独立会话”思路。

#### 场景 D：天气 / 汇率 / 新闻 / K 线快照

- **机制**：Cron(isolated)
- **例子**：
  - 每天 9:00 推送北京天气
  - 每小时推送 BTC/ETH/股指价格快照
  - 每天早上 8:30 推送科技要闻 5 条
- **特点**：
  - 每条消息都是“自包含”的结果，不需要“上次”概念
  - 独立会话，不污染主会话聊天历史

#### 场景 E：资源用量报表

- **机制**：Cron(isolated)
- **例子**：
  - 每天 7:00 生成服务器资源用量报表（CPU/内存/磁盘）
  - 每天 8:00 总结云账单 / API 消耗情况
- **实现建议**：
  - Cron(isolated) 会话中：
    1. 调接口拉原始数据
    2. 用 Agent 写一份人读得懂的简短总结
    3. 发送给你 / 或写入一个 `reports/` 目录（如你希望避免刷屏）

#### 场景 F：周期性清理任务（脚本 + 简短汇报）

- **机制**：Cron(isolated)
- **例子**：
  - 每天 3:00 清理本地缓存目录、临时文件
  - 每周日 4:00 做一次日志压缩归档
- **特点**：
  - 执行结果通常可以只写简短一句：
    - “已清理 X MB 临时文件”
  - 不需要“历史记忆”，只做眼前操作即可

---

### 4.3 Cron(main) 场景（有记忆的定点任务）

这些任务既有明确的固定时间点，又非常依赖上下文、长期记忆。

#### 场景 G：每日工作回顾 + 明日计划

- **机制**：Cron(main)
- **触发**：每天 23:00
- **行为**：
  1. 从当天的 `memory/2026-02-23.md`、聊天记录中梳理：
     - 完成了什么任务
     - 有什么 open issues
  2. 写一份：
     - 今日回顾（3～5 条）
     - 明日计划建议（3～5 条）
- **特征**：
  - 强烈依赖主会话历史：你今天和 Agent 聊过什么、做了什么
  - 适合写入 `learnings/YYYY-MM-DD.md` 再同步到 `MEMORY.md`

#### 场景 H：周报 / 月报辅助生成

- **机制**：Cron(main)
- **触发**：
  - 每周五 17:00 → 周报草稿
  - 每月最后一天 18:00 → 月报草稿
- **行为**：
  - 收集最近一周/一月的 `memory/*.md`、项目进展、异常报告
  - 汇总成一份结构化 Markdown 草稿：
    - 本周完成事项
    - KPI / 指标
    - 风险与问题
    - 下周计划建议
- **优势**：
  - 主会话中长期积累的信息，在这个时间点被转换为“有价值文档”
  - 你再人工微调即可对外发送

#### 场景 I：学习总结 / 读书笔记整理

- **机制**：Cron(main)
- **例子**：
  - 每晚 22:30：
    - 搜索今天对话中所有与“学习”、“阅读”、“笔记”相关的段落
    - 提炼为 `learnings/YYYY-MM-DD.md` → 更新长期 `MEMORY.md`
  - 每周末：
    - 生成一份“本周学习成果总结”

---

### 4.4 混合模式场景（Heartbeat + Cron 配合）

#### 场景 J：项目“持续健康监控” + “固定时间详细报告”

- **组件**：
  - Heartbeat：频繁检查，异常立刻通知
  - Cron(isolated)：每天固定时间生成详细项目状态报告
- **例子**：
  - Heartbeat：
    - 每 10 分钟检查 CI 状态 / 错误率
    - 异常时发告警 + 写入 `memory`
  - Cron(isolated)：
    - 每天 18:00：
      - 拉所有项目状态、CI 历史、错误统计
      - 生成一份日报（放文件 or 发消息）

#### 场景 K：邮件重点筛选 + 每日汇总

- **组件**：
  - Heartbeat：每隔一段时间扫描新邮件，遇到高优先级邮件实时提醒
  - Cron(main)：每天 20:00 给你一个“今日邮件 Digest”
- **逻辑**：
  1. Heartbeat：
     - 根据 `USER.md` + 你最近聊的指示（例如“本周关注融资邮件”），筛选出关键邮件
     - 实时提醒 + 写日志
  2. Cron(main)：
     - 基于今日 memory，总结“今天收到的重要邮件 + 你是如何处理的 + 还有什么没处理”

## 5. 实操建议：如何在你自己的环境里落地这些模式

> 这部分尽量给可操作的“套路”，但 CLI 精确命令需结合你当前版本的 `openclaw --help` 校验。没有文档支撑时，我不会给具体命令。

### 5.1 Heartbeat：写好 HEARTBEAT.md 是关键

1. 打开你的工作空间目录（通常在 `~/.openclaw/workspace`，可通过配置自定义）
2. 创建或编辑 `HEARTBEAT.md`：
   - 明确写：
     - 你要监控什么（项目列表 / 服务列表 / 邮箱列表）
     - 通知条件（异常阈值）
     - 行为规范：正常只返回 `HEARTBEAT_OK`，异常时如何汇报
3. 检查日志，确认 Heartbeat 真在跑：
   - 观察 logs/ 目录中是否定期有 heartbeat 相关日志
   - 如果没有，需要检查 Scheduler 是否启用（查配置文件中的 `heartbeat`/`scheduler` 段）

### 5.2 Cron(isolated)：让 Agent 显示“准备调用参数”，你来审核

- 在主会话中发指令，示例：

> “请为我创建一个 Cron 任务：  
> - 每 15 分钟在独立会话中运行  
> - 内容：检查 VideoBuilder 项目的 CI 状态，如果有失败则汇报给我，否则简单汇报当前通过情况  
> 请先给出你要调用 `cron` 工具的完整参数 JSON，我要检查后再让你真正调用。”

- 根据 TOOLS 描述，确认其中是否有类似：
  - `schedule`: `"*/15 * * * *"`
  - `sessionTarget`: `"isolated"`
  - `message` / `payload.message`: 任务说明文本
- 确认无误后再让 Agent 真正调用 `cron` 工具。

> 这样你可以“人肉审核” Cron 配置，避免 Agent 搞出一堆你不想要的任务。

### 5.3 Cron(main)：把任务当成“给主会话发一条系统消息”

概念上你可以这样理解 Cron(main) 的配置：

- `sessionTarget`: `"main"` 或不写（使用默认主会话）
- Payload 描述类似：
  - `"kind": "agentTurn"`
  - `"message": "请你基于最近一周的 memory 日志，生成一份周报草稿。"`

由于官方文档没有统一列出所有字段名，不建议我在这里给一个“看起来就能执行”的命令行示例——那会很容易误导。  
更稳妥的做法还是：在你当前版本里查看 `TOOLS.md` 中 `cron` 工具定义，或在对话里让 Agent 展示它准备调用的 JSON。

## 6. 一句话记忆口诀（方便你脑内快速切换）

- **在会话里** = 能看到你之前聊过什么 + 读取记忆文件
- **在会话外** = 每次都是“陌生人”，只做眼前配置的事

对应到三种机制：

- **Heartbeat**：  
  → 永远在主会话、共享记忆  
  → 偏“后台守护 + 异常才说话”

- **Cron(main)**：  
  → 在主会话里按时间点触发  
  → 适合“有记忆 + 有固定时间”的任务（周报、日总结）

- **Cron(isolated)**：  
  → 每次新建独立会话、执行完就销毁  
  → 适合“报表类、查询类、固定流程、不需要记忆”的任务（你现在的项目 15 分钟汇报、天气等）
